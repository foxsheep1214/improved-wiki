# 塞防大硬件管理 — 大硬件总负责人工作手册

胡杨（大硬件总负责人，统管 数字硬件/结构/天线射频/测试硬件）的日常管理工作手册：实验室规划、团队人力、项目跟进、供应商管理、OKR、汇报文档、管理思想记录。**知识库/datasheet/元器件库建设见 `hardware-designer` 技能。**

## ⚠️ 铁律（每次回复前自检，全文见 `references/工作纪律.md`）

1. **结论先行**：先给结论/答案，再说原因。需要行动就给 1/2/3/4，不要"首先然后最后"。
2. **数据精准**：数量/日期/百分比必须准确，不确定的用"预估/约"标注。
3. **先查后试**：用户问"X 怎么做/有没有标准方式" → 先 grep skill 文档 + 找现成 API/工具，再动手；不要连试 5+ 次 silent no-op 才回头查。
4. **元问题不凭印象**："X 是不是做了/进度如何" → 先查文件 mtime + session_search + 状态文件，再答；查不到就说"找不到记录"，别说"没做"。
5. **不主动关联项目**：用户没提 Tracker II 就别扯它，即使技术重叠。问 A 答 A。
6. **改前先备份**：改文档/代码前 `cp 原文件 ~/.Trash/...bak`；删文件用 trash 不彻底删。
7. **文件放桌面**：报告/文档完成后放桌面让胡杨自己 Review，不主动描述内容。

## ⚠️ 实验室体系定位（写实验室文档时先于一切）

公司含**五个专业实验室**，硬件实验室只是其一，定位须放在全体系背景下，不能 standalone。文档第一节必须先写"实验室体系定位"交代五室分工，再聚焦硬件实验室，避免把五室职责全塞进硬件实验室章节。

| 实验室 | 主管部门 | 核心职责 |
|--------|---------|---------|
| 硬件实验室 | 智能硬件部 | 模块级数字硬件来料/出货检验、制造加工、功能性能测试、系统级联调、技术/工艺定型、生命周期维护 |
| 射频实验室 | 智能硬件部/天线射频部 | 射频链路调试、天线测试、功放性能验证 |
| 结构实验室 | 结构部 | 结构件验证、工装夹具测试 |
| 可靠性实验室 | 测试部 | 环境试验、可靠性加速老化、失效分析 |
| 系统测试实验室 | 测试部 | 系统级功能性能测试、集成测试 |

### 2. 团队与人力（当前约束）
- layout 人力（郑细兰）是单点瓶颈：指定 1 名后备结对、单人 ≥3 块板触发委外；米亚松 5 月离职，结构部紧张，接手人写入 `研发人力资源.md`。
- 关键岗位统一后备原则；新增工程师同步 `任务完成情况.md`（建档）+ `研发人力资源.md`（入职日期）。

### 3. 项目与研发管理
- **任务文件分离（不可混用）**：`任务-研发资源.md` = 任务维度；`产品及项目管理.md` = 项目维度（项目名/代号/状态/负责人）。
- **任务表标准列（每条任务必含）**：任务 / 负责人 / 优先级(P0·P1·P2) / 开始 / 计划完成 / 实际完成 / 状态 / 阻塞原因 / 评审。
- **日期纪律**：新增立即填开始日期；完成立即填实际完成并归档 `任务完成情况.md`。不事后补录。
- **WIP 上限**：个人在研 P0/P1 ≤ 2 项；新需求先入表标"待排"，周会统一定级，不接受口头插队。
- **评审门禁**：原理图冻结、layout 送样/委外前各做一次 checklist 互审，未过不下单（SOP 见 `references/组内互审SOP.md`）。
- **TR 门禁**：每个 TR（TR4/4A/5）预先定硬件出口条件，达标才进下一阶段。
- **例会节奏**：周会（进展/难点/支持，看 WIP·准时率·缺陷回退）；月度复盘（准时率+缺陷+风险闭环）。
- **patch 注意**：总览文档每次 patch 只操作一行，避免覆盖相邻行。
- 研发状态流：计划 → TR3 → TR4 → TR4A → TR5 → 转产 → 生命周期迭代。

### 4. 供应商与采购
- **台账实际位置**：`~/Documents/供应商对接记录/`（`供应商台账.md` 索引 + 每供应商子目录含 `供应商信息.md` + 历次对接记录）。编辑规范/模板/分类/打分卡见 `references/供应商对接操作手册.md`。
- **器件选型铁律**：候选 **≥2 家**（含二供/国产替代），核查出口管制（EAR/实体清单），输出对比表后再定点。**单一来源器件**登记到下方风险登记册。

### 5. OKR 管理（数字硬件部门）
- **顺序铁律**：先从 `任务-研发资源.md` 列出部门承接的全部任务清单，**再**基于清单起草 OKR。❗不要跳过列任务直接写 OKR——胡杨会纠正。
- **结构**（多轮验证）：O1 保障数字硬件研发任务｜O2 硬件实验室建设与规范化｜O3 数字硬件 Auto-Development 启动。（不加 O4，供应链风险非本季度重点。）
- **迭代节奏**：胡杨倾向"逐轮暂停确认"。信号：「暂停确认效果」→ 等待；「可以，继续」→ 下一轮；「整理好吧」→ 停止并汇总。

### 6. 汇报文档
- **工具**：pandoc + Playwright(Chrome) 生成 PDF，**不用 Marp**（Marp 定位幻灯片，不适合正式技术报告）。
- **8 页结构**：封面 → 部门定位 → TOP3 → 根因 → 近期行动(4 点纯文本) → 需要的支持(授权/人/测试资源) → 远期目标(讲"持续改进"不讲 1/2/3 年) → 总结。逐页确认，不一次全量展示。
- **命名**：`{题目} - {YYYYMMDD} - {作者}.md`（如 `硬件Auto-Development路径 - 20260603 - 胡杨.md`）。
- 战略探索类汇报（Auto-Development 等）8 阶段+多 Agent 框架见 `references/hardware-design-8-stage-workflow.md`。

### 7. 管理思想记录
- 必读 `~/Documents/大硬件管理/管理思想.md`；读写规范见 `references/管理思想操作手册.md`。
- 新增语录：保留原文引述 + 注明出处（会议/日期/发言人）+ 附 3-5 条要点解读。
- patch 前必先 read 全文：同一语录可能同时出现在正文和附录两处。
