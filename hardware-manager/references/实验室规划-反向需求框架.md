# 硬件实验室规划文档框架

> 适用场景：按照公司要求（5月5日18点前）输出实验室落地计划
> 基于公司通知模板：方案落地计划（含落地时间线/关键里程碑/资源需求/风险预判）+ 对应需求点（业务问题/战略关联/预期收益）
> 最后更新：2026-05-16

---

## 文档框架

### 框架一：实验室定位

一句话说明实验室在研发体系中的位置，服务于哪些产品线，承上启下关系。

### 框架二：落地计划时间线（三阶段）

按时间分阶段，每阶段有里程碑 + 负责人 + 说明。

**⚠️ 常见错误**：先写"我们要做什么"（设备采购/工位划分），再套"业务问题"。胡杨要求反过来——先定义问题域，再推导落地动作。

### 框架三：需求点（核心部分）

每个需求点必须包含三要素，按此顺序展开：

| 要素 | 内容 |
|------|------|
| 解决的业务问题 | 当前痛点是什么（具体到人和具体现象） |
| 与公司战略的关联 | 为什么这个问题影响公司战略 |
| 预期价值/收益 | 量化收益（减少改版频次/缩短周期/节省成本） |

**⚠️ 常见错误**：把"资源需求"（要人/要钱）当作需求点。资源需求是支撑内容，不是核心问题本身。

### 框架四：资源需求

列出人力/场地/设备/费用需求，标注优先级和到位周期。

### 框架五：风险预判

每个风险点：影响 / 概率 / 应对措施。

### 框架六：战略关联性

表格形式，说明实验室能力建设与公司产品战略（Tracker II / Thunder / 六条产品线量产能）的对应关系。

### 框架七：后续管理机制

周度跟踪 + 月度复盘 + 管理工具（飞书任务）。

---

## 关键教训

- **反向框架**：先问题→后方案，不要先方案→后问题
- **需求点 ≠ 资源需求**：前者讲业务价值，后者讲具体人手/设备
- **时间线要可检查**：里程碑带日期和负责人，方便周度跟踪
- **Thunder 5月7日节点**是高风险项，实验室规划中应体现优先级保障
