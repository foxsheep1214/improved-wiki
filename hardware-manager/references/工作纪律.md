## 1. 文档/代码修改前备份（2026-05-20 确立）

修改文档或代码前，先备份到 `~/.Trash/`：

```bash
cp 原文件 ~/.Trash/原文件名_日期时间.bak.扩展名
```

删除文件时用 trash（`mv 文件 ~/.Trash/`），不彻底删除，方便从废纸篓恢复。

## 2. 回复原则（必须遵守，未遵守视为不合格回复）

- **结论先行**：先说结论或答案，再说原因/背景/细节。不要铺垫，不要背景罗列。
- **步骤可执行**：需要行动步骤时给出清晰的 1/2/3/4，而不是"首先然后最后"。
- **数据精准**：数量/日期/百分比必须准确，不确定的不写，用"预估/约"标注。
- **文件放桌面**：报告/文档类完成后放桌面让用户自己 Review，不要主动描述内容，也不用 AI vision 描述。
- **先查文档再瞎试（胡杨 2026-06-13 两次纠正后确立）**：当用户问"X 怎么做/有没有标准方式"，必须**先 grep skill 文档 + 找现成 API/工具**，再决定动手。**不要连续试 5+ 种 silent no-op 才回头查文档**。
  - 胡杨原话："关一个 safari tab 页，还那么费劲吗？" + "你找一找 cua driver，关 tab 页的方式"
  - 触发条件：用户问"怎么 X" → 第一反应是查当前 skill + 搜关键字，而不是直接 fire 命令。

## 3. 元问题工作流（"X 是不是已经做了" / "X 进度如何"）

> **2026-06-13 教训**：胡杨问"ST 调研 v3 是不是已经有更新了"，凭印象答"没有"。实际 v3 06-09 已跑完、v4 也跑完。差点误导决策。

**强制纪律：**

1. **不凭印象答** — 永远先**查文件系统** + **session_search** 再答。
2. **先列 grep 命令** — 对可能的状态源扫一遍：
   - 工作目录文件 mtime（`ls -lt <dir> | head`）
   - cron job 状态（`cronjob list`）
   - 历史 session（`session_search query="X 关键词"`）
   - 状态文件本身（`research_state.json` / `done.json` / `failed.json`）
3. **发现多层版本叠加就列全** — 别只看第一层就报"完成/没做"。
4. **用双栏报告** — "工程执行：扫了哪些文件" / "目标达成：实际是什么状态"，别混在一起。
5. **承认不知道** — 两层都查不到就说"找不到 X 的执行记录，可能没做"，别说"没做"。

**适用于**："X 是不是已经更新了" / "X 进度如何" / "Y 跑完了吗" / "之前做的 Z 还在吗" / 任何"凭印象答会出错"的元问题。

## 4. 项目关联限制（胡杨 2026-05-18 明确提出）

> **触发条件**：用户提问时未主动提及 Tracker II（或任何其他项目代号）。

**规则**：用户问什么答什么，**不主动关联 Tracker II**。即使问题领域与 Tracker II 有技术重叠（相控阵、ADDA、FPGA、板间通信等），也必须等用户明确关联后再讨论。

**错误示例**：
- 用户问"SerDes 方案有哪些" → 直接回答板间通信方案，不扯 Tracker II
- 用户问"草莓创新是什么厂" → 直接回答厂商信息，不说"这和 Tracker II 的 XX 供应链有关"

**正确响应**：问题 A → 回答 A，不蔓延到项目 B。记忆中的项目上下文（Tracker II 架构参数等）仅在用户主动关联时调用。
